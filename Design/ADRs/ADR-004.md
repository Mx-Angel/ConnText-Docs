# ADR-004: MLS via OpenMLS as Unified E2EE Protocol

## Status
Accepted (supersedes ADR-002)

## Context
ADR-002 selected libsignal for 1-to-1 encryption and Megolm for group encryption but several issues emerged:

- libolm (the original Megolm implementation) was deprecated by Matrix in 2024 due to memory safety CVEs in its C/C++ code.
- vodozemac is the Rust successor to libolm, but still implements the same Olm/Megolm protocols which require two separate mechanisms for 1-to-1 and group conversations.
- ADR-003 moved the SDK to Rust, making Rust-native crypto libraries the natural choice.

## Decision
ConnText will use MLS (Messaging Layer Security, RFC 9420) as its sole E2EE protocol for both 1-to-1 and group conversations, implemented via the OpenMLS library.

## Alternatives Considered
- **libsignal + Megolm (ADR-002 original):** Possible, but not ideal due to the complexity of managing two different protocols on top of the different languages.
- **vodozemac (Olm + Megolm):** Viable Rust implementation, but requires managing two separate protocols.
- **libsignal only:** Excellent for 1-to-1 but scales poorly for groups.
- **Custom protocol using libsodium primitives:** Maximum flexibility, but designing a cryptographic protocol is high-risk even for experts.

## Consequences
- Single protocol for all conversation types simplifies the Encryption Engine component.
- MLS handles group key management with O(log n) scaling, efficient for large channels.
- OpenMLS is Rust-native, integrating directly with the SDK (ADR-003) without FFI overhead.
- OpenMLS mobile targets (Android, iOS) are built but not fully tested on CI: accepted as low risk given mobile clients are years away.
