# ADR-006: Encrypted Key Backup for Multi-Device Support

## Status
Accepted

## Context
ConnText supports multi-device access, so users may log in from a desktop, laptop, and phone. With MLS encryption (ADR-004), decryption keys only exist on the device that participated in the conversation when messages arrived.

Without a key transfer mechanism, logging in on a new device means the user cannot decrypt any prior message history. All channels would appear empty despite containing years of conversation. This presents as a critical bug to users even though the encryption is working correctly.

NFR-PRIV-09 (sessions never sync across devices) has been removed as it directly contradicted multi-device support.

## Decision
ConnText will implement encrypted key backup. The user's MLS key material is encrypted client-side using a recovery passphrase, then uploaded to their home server as an opaque blob. The server stores it but cannot decrypt it.

On a new device, the user logs in and enters their recovery passphrase. The SDK downloads the encrypted key bundle, decrypts it locally, and restores the ability to decrypt historical messages.

## Alternatives Considered
- **No history on new devices:** New devices start fresh and can only decrypt messages received after joining. Simple and maximally private, but terrible UX. Signal used this approach for years and users consistently reported it as a bug.
- **Device-to-device key transfer:** The old device directly transfers keys to the new device (e.g. via QR code or local network). Avoids server involvement entirely, but requires the old device to be available. Doesn't help if the old device is lost, stolen, or broken.

## Consequences
- The home server must store encrypted key backup blobs.
- The SDK's Encryption Engine handles backup creation, encryption with recovery passphrase, and restoration.
- The Auth/onboarding flow must prompt for recovery passphrase on new device login.
- If the user forgets their recovery passphrase and loses all devices, historical messages are permanently unrecoverable, this is by design, not a bug.
- The recovery passphrase is never sent to the server.
- Key backup does intentionally weaken MLS forward secrecy by preserving historical key material, but this is a deliberate tradeoff for multi-device usability.
