# ADR-003: Rust as SDK Implementation Language

## Status
Accepted

## Context
ConnText's SDK contains all core business logic, including E2E encryption. The E2E encryption ecosystem has moved almost entirely to Rust: OpenMLS, libsignal, vodozemac, etc. are all Rust-native. Using a non-Rust SDK language would require cross-language FFI specifically for the most security-critical component.

## Decision
The SDK will be written in Rust. The C++ Qt desktop client will call the SDK through a C FFI boundary. Mobile and other platform bindings will be generated using UniFFI.

## Alternatives Considered
- **C++ SDK with Rust crypto via FFI:** Keeps C++ as the primary language but requires managing a cross-language boundary in the encryption layer which is the most security-critical part of the codebase. Two build systems (CMake + Cargo) to maintain.
- **Full C++ SDK with C++ crypto libraries:** Avoids cross-language complexity entirely, but the available C++ crypto implementations (libolm) are deprecated due to memory safety vulnerabilities and libsignal is written in Rust. Would require using less maintained or untested libraries.

## Consequences
- Build system changes from CMake to Cargo for the SDK.
- Binding generation uses UniFFI for Kotlin, Swift, and Python; C FFI for the C++ desktop client.
- Rust must be learned before SDK development begins.
- Memory safety is enforced by the compiler in all SDK code, including the encryption layer.
- The crypto ecosystem (OpenMLS, rustls, tokio) integrates natively without FFI overhead.
- References ADR-002 (E2EE decision) as a primary driver.
