# ADR-005: Per-Channel E2EE with Opt-In at Creation

## Status
Accepted

## Context
NFR-SEC-01a mandated E2EE for all messages, but two backend capabilities assume the server can read plaintext:

1. The RAD register lists a server-side indexing service for message search but the server cannot index ciphertext.
2. Virus scanning was listed as a mitigation for file uploads but the server cannot scan encrypted files.

Enforcing E2EE everywhere means losing both. Removing E2EE to keep them contradicts ConnText's privacy-first goals.

## Decision
E2EE is configured per channel at creation time. The channel creator chooses whether to enable E2EE, presented with a clear explanation of the features lost. Once set, the encryption mode is permanent and cannot be changed.

Standard encrypted channels (TLS only): search indexing, link previews, and upload scanning are available.

E2EE channels: the server sees only ciphertext. Search runs against the SDK's local SQLite index, and server-side scanning is unavailable. The SDK will handle media processing before encryption for E2EE channels.

## Alternatives Considered
- **E2EE everywhere, no exceptions:** Maximum privacy, but eliminates search and scanning. Users who need those features have no option.
- **No E2EE, TLS only:** Simplest implementation, but contradicts ConnText's privacy-first positioning and provides no protection against a compromised server.
- **Default unencrypted, opt-in E2EE:** Viable but undermines the project's privacy stance.

## Consequences
- NFR-SEC-01a updated from "all messages" to "per-channel at creation".
- Channel creation API requires an encryption mode field.
- Encryption mode is immutable after creation to prevent false security (can't retroactively encrypt plaintext history or decrypt E2EE history).
- The SDK must handle local search indexing for E2EE channels.
- UI must clearly indicate which mode a channel is in at all times.
