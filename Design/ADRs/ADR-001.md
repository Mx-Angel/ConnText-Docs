# ADR-001: Home Server Identity Model

## Status
Accepted

## Context
ConnText needs an identity model that allows users to have a single account that works across multiple self-hosted servers, without requiring users to manage cryptographic keys manually or create separate accounts on every server.

## Decision
ConnText will use a home server identity model. Each user registers on one server (their "home server"), and that server is the authority for their identity. When the user connects to other servers, those servers verify the user's identity with their home server.

## Alternatives Considered
- **Client-side keypair identity:** User generates a keypair locally, presents public key to any server. Rejected because losing the private key means losing the account entirely, putting too much burden on non-technical users.
- **Per-server accounts:** Users register separately on each server. Rejected because it fragments identity — five servers means five accounts with no unified presence.

## Consequences
- Servers need a federation protocol to verify visiting users.
- DMs can route through the recipient's home server, solving the DM routing problem.
- A user's home server is a single point of failure for their identity.
- Need to consider a migration/export mechanism for when a home server shuts down permanently.
